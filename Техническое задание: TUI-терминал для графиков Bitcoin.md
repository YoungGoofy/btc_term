# Техническое задание: TUI-терминал для графиков Bitcoin (USDS-M Futures)

## 1. Общее описание
Разработка высокопроизводительного консольного TUI-приложения на Go для отображения графиков деривативов Binance (фьючерсов USDS-M) в реальном времени. Приложение должно строить свечи Heikin Ashi, отображать набор технических индикаторов в изолированных панелях и поддерживать мгновенное переключение таймфреймов (1m, 5m, 15m, 4h). 

## 2. Стек технологий и среда разработки
* **Язык:** Go 1.21+.
* **Ядро UI (Архитектура):** `github.com/charmbracelet/bubbletea`. Строгое использование паттерна Model-View-Update (MVU). Это критически важно для работы с асинхронными вебсокетами без блокировок и гонок данных.
* **Стилизация (Layout):** `github.com/charmbracelet/lipgloss` для построения жесткой сетки панелей и границ.
* **Графика:** Кастомный рендерер свечей на базе символов Брайля для обеспечения максимального разрешения (позволяет накладывать линии индикаторов прямо поверх свечей в терминале).
* **Индикаторы:** `github.com/markcheno/go-talib` для расчета RSI, MACD и EMA. Логика для VWAP, Heikin Ashi и Pivots HL 10 реализуется разработчиком вручную для инкрементального пересчета.
* **Сеть:** `github.com/gorilla/websocket` для работы со стримами Binance.

## 3. Пользовательский интерфейс (UI)
Интерфейс разделен на три фиксированные горизонтальные панели, занимающие всю доступную высоту окна терминала.

* **Панель 1 (Верхняя, ~50% высоты):** * График цены **BTCUSDT** (отрисовывается в формате свечей Heikin Ashi).
  * Линия индикатора **VWAP**, наложенная поверх графика цены.
* **Панель 2 (Средняя, ~25% высоты):**
  * Линии **EMA 9** и **EMA 21**.
  * Индикатор **Pivots HL 10**. 
  * *Требование:* Поскольку эти индикаторы привязаны к абсолютной цене, ось Y в этой панели динамически масштабируется под их текущие значения.
* **Панель 3 (Нижняя, ~25% высоты):**
  * **MACD** (Гистограмма, линия MACD, линия Signal).
  * **RSI**.
  * *Требование:* Так как MACD колеблется около нуля, а RSI строго от 0 до 100, необходимо разделить эту панель на две колонки (`lipgloss.JoinHorizontal`), либо выводить две независимые оси Y по краям одной панели.

**Навигация:** Всплывающее модальное окно (можно использовать готовый компонент `bubbles/list`) для выбора таймфрейма (1m, 5m, 15m, 4h). Окно должно вызываться по горячей клавише (например, `Tab` или `F`) и перекрывать основной интерфейс.

## 4. Источник данных: Binance USDS-M Futures API
Интеграция состоит из двух обязательных этапов для каждого таймфрейма: загрузка исторического "среза" и подхват живого потока.

### 4.1. REST API (Исторические данные для "прогрева")
Для корректного расчета EMA и MACD на текущей свече требуется история предыдущих значений.
* **Endpoint:** `GET https://fapi.binance.com/fapi/v1/klines`
* **Параметры:** `symbol=BTCUSDT`, `interval={таймфрейм}`, `limit=500`.

### 4.2. WebSocket API (Живой поток)
* **Base URL:** `wss://fstream.binance.com/ws`
* **Stream:** `<symbol>@kline_<interval>` (например, `btcusdt@kline_1m`). 
* *Внимание:* Символ в URL вебсокета должен передаваться строго в нижнем регистре. Соединение необходимо поддерживать (отвечать на ping-фреймы сервера).

**Структура полезной нагрузки (Payload) для десериализации:**
```go
// Основное сообщение вебсокета
type BinanceWSMessage struct {
	Event string         `json:"e"` // Тип события (ожидаем "kline")
	Time  int64          `json:"E"` // Время события
	Kline BinanceWSKline `json:"k"` // Данные свечи
}

// Вложенный объект свечи
type BinanceWSKline struct {
	StartTime int64  `json:"t"` // Начало свечи
	EndTime   int64  `json:"T"` // Конец свечи
	Open      string `json:"o"` // Цена открытия (нужна конвертация в float64)
	High      string `json:"h"` // Максимум
	Low       string `json:"l"` // Минимум
	Close     string `json:"c"` // Закрытие
	Volume    string `json:"v"` // Объем торгов
	IsClosed  bool   `json:"x"` // Флаг закрытия свечи
}
```

## 5. Жизненный цикл и потоки данных (Интеграция с MVU)
1. **Инициализация или смена таймфрейма:**
   * Блокировка UI (показ спиннера загрузки).
   * Отправка команды (через `tea.Cmd`) на REST API для получения 500 свечей.
   * Конвертация сырых свечей в массив структур Heikin Ashi.
   * Первичный расчет всех индикаторов по историческому массиву.
2. **WebSocket Подписка:**
   * Открытие WS-соединения. Цикл чтения (ReadMessage) выносится в отдельную горутину, которая возвращает кастомные сообщения (например, `TickMsg`) в главный цикл `Bubble Tea`.
3. **Обновление состояния (Update):**
   * При получении `TickMsg` программа проверяет поле `StartTime` пришедшей свечи. 
   * Если время совпадает с последней сохраненной свечей — обновляем её значения (на лету пересчитывая хвосты индикаторов). 
   * Если время больше — добавляем новую свечу в конец массива, удаляя самый старый элемент в начале (Shift), чтобы предотвратить переполнение памяти.
4. **Отрисовка (View):**
   * Модель возвращает обновленную строку интерфейса. `lipgloss` собирает элементы из матриц графиков и выводит их на экран без мерцаний.
